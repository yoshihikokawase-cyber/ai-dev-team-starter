# 受け入れ基準と実装完了判定

このドキュメントは、機能が「完成」したと判断するための基準を定義します。
すべてのエージェントはこの基準に従って完了判定を行ってください。

---

## 完了の定義（Definition of Done）

以下の条件をすべて満たしたとき、タスクは「完了」とみなします:

1. **仕様書の受け入れ条件が全て満たされている**
2. **自動テストが全て通過している**（既存テスト + 新規テスト）
3. **手動確認または探索的テストが完了している**
4. **High重大度のバグが0件である**
5. **ドキュメントが更新されている**（必要な場合）
6. **コードがレビュー可能な状態である**（過剰な未使用コードがない）

---

## 実装完了判定チェックリスト

### 仕様・要件の確認
- [ ] 仕様書（`docs/requirements/`）の機能要件を全て実装した
- [ ] 仕様書の非機能要件を満たしている
- [ ] 受け入れ条件を全て満たしている
- [ ] スコープ外の機能は実装していない

### 実装の品質確認
- [ ] ビルドが成功する（`npm run build` 等）
- [ ] 型チェックが通る（TypeScriptの場合）
- [ ] リントエラーがない（`npm run lint` 等）
- [ ] 既存の自動テストが全て通過している
- [ ] 新機能のテストコードを追加した（またはPlaywrightテストが実行済み）

### セキュリティ確認
- [ ] ユーザー入力をHTMLにそのまま出力していない（XSS対策）
- [ ] SQLクエリにユーザー入力を直接埋め込んでいない（SQLインジェクション対策）
- [ ] 認証・認可が必要な箇所に適切なチェックがある
- [ ] 機密情報（APIキー・パスワード）をコードにハードコードしていない

### テスト完了確認
- [ ] Playwright正常系テストが全て通過している
- [ ] Playwright異常系テストが全て通過している
- [ ] 境界値テストが実施されている
- [ ] 探索的テスト（Antigravity）が実施されている
- [ ] High重大度のバグが0件である
- [ ] Medium重大度のバグが解消されているか、次フェーズへの持ち越しが承認されている

### ドキュメント確認
- [ ] 仕様書が実装内容と一致している
- [ ] 設計変更があれば `docs/decisions/` に記録した
- [ ] APIや主要な関数の変更があれば `docs/architecture/overview.md` を更新した

## 品質基準（数値目標）

### パフォーマンス基準（デフォルト）
| 指標 | 基準値 |
|---|---|
| ページ初期表示 | 3秒以内（通常のネットワーク環境） |
| APIレスポンスタイム | 1秒以内（P95） |
| Largest Contentful Paint | 2.5秒以内 |

_プロジェクト固有の要件がある場合は上書きしてください_

### テストカバレッジ基準（推奨）
| 種別 | 基準 |
|---|---|
| ユニットテスト | コアロジックの80%以上 |
| E2E（Playwright） | 主要ユーザーフローを100%カバー |
| 探索的テスト | リリース前に毎回実施 |

## 受け入れ条件の書き方テンプレート

新機能の受け入れ条件は以下の形式で記述してください:

```
Given [前提条件・初期状態]
When  [ユーザーが行う操作]
Then  [期待する結果・状態変化]
```

### 良い例
```
Given ユーザーが登録フォームを開いている
When  有効なメールアドレスと8文字以上のパスワードを入力して「登録」ボタンをクリックする
Then  アカウントが作成され、ダッシュボードページへリダイレクトされ、「登録が完了しました」というメッセージが表示される
```

### 悪い例
```
When ユーザーが登録する
Then 成功する
```
（操作・期待結果が具体的でないためテスト不可能）

## バグの優先度別対応方針

| 重大度 | 対応方針 |
|---|---|
| **High** | リリースブロック。修正後に再テストが必須 |
| **Medium** | リリース前の修正を推奨。修正しない場合は責任者の承認が必要 |
| **Low** | 次のスプリントで修正。バックログに追加 |
