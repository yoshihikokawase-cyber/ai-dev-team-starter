# テスト運用ガイド

---

## テストの基本方針

このプロジェクトでは3層のテストを組み合わせて品質を担保します:

| テスト種別 | ツール | 目的 | 実施タイミング |
|---|---|---|---|
| ユニットテスト | Jest / Vitest 等 | ロジックの正確性 | 実装中・PR前 |
| E2Eテスト | Playwright | ユーザー操作フローの自動検証 | PR前・デプロイ前 |
| 探索的テスト | Google Antigravity | 想定外操作・UI崩れ・バグ探索 | リリース前 |

---

## Playwrightの位置づけと運用

### 目的
Playwrightは「仕様書の受け入れ条件」を自動テストとして表現するためのツールです。
人が手動で確認していた正常系・異常系・境界値テストを自動化します。

### テストの作成ルール
- テストファイルは `tests/e2e/` に配置する
- ファイル名は `[機能名].spec.ts` とする
- セレクターは `data-testid` 属性を優先する
- 固定待機（`waitForTimeout`）は使わない

### 実行方法

```bash
# 全テスト実行
npx playwright test

# 特定のテストファイルだけ実行
npx playwright test tests/e2e/user-registration.spec.ts

# UIモードで実行（デバッグに便利）
npx playwright test --ui

# ヘッドありモードで実行（ブラウザを表示する）
npx playwright test --headed

# レポートを表示
npx playwright show-report
```

### テスト構成の推奨

```typescript
test.describe('[機能名]', () => {
  test.describe('正常系', () => { /* ... */ });
  test.describe('異常系', () => { /* ... */ });
  test.describe('境界値', () => { /* ... */ });
});
```

詳細は `skills/create-playwright-test/SKILL.md` を参照してください。

## Google Antigravityの位置づけと運用

### 目的
Antigravityは「Playwrightで書ききれない想定外の操作」を探索するためのツールです。
リリース前に必ず実施し、自動テストでは見落とすエッジケースを発見します。

### 実施タイミング
- リリース前（必須）
- 大きな変更後（UI変更・機能追加）
- バグ修正後（回帰確認として）

### 探索の観点

1. **想定外操作**: ランダムクリック・高速連打・ブラウザ戻るの乱用
2. **異常入力**: 空文字・超長文・特殊文字・SQLインジェクション相当
3. **UI崩れ**: 複数のブラウザサイズでのレイアウト確認
4. **データ不整合**: リロード・中断・二重送信などの操作

詳細は `skills/exploratory-test-antigravity/SKILL.md` を参照してください。

## 不具合報告テンプレート

バグを発見した場合は必ず以下のフォーマットで記録してください:

---

```markdown
## バグ報告 #[連番]: [バグの概要タイトル（30文字以内）]

**日時**: YYYY-MM-DD HH:MM
**発見者**: [発見したエージェントまたは担当者]
**対象機能**: [バグが発生している機能名]
**重大度**: High / Medium / Low

---

### 発生条件

| 項目 | 内容 |
|---|---|
| ブラウザ | Chrome XX / Firefox XX / Safari XX |
| OS | Windows 11 / macOS 14 / iOS 17 |
| 画面サイズ | XXXpx × YYYpx |
| ユーザー状態 | ログイン済み / 未ログイン / [特定の状態] |
| テストデータ | [使用したデータの内容] |

### 再現手順

> この手順を読んだ人が同じ操作をすれば100%再現できるレベルで書くこと

1. [URLを開く: http://localhost:3000/XXX]
2. [XXXフィールドに「YYY」を入力する]
3. [「送信」ボタンをクリックする]
4. [XXXページを確認する]

### 期待する動作

[仕様書に基づいた、本来あるべき挙動を具体的に書く]

### 実際の動作

[実際に起きていること。エラーメッセージがあればそのまま貼り付ける]

### スクリーンショット・ログ

[スクリーンショットのパスまたは、コンソール・ネットワークエラーのログを貼り付け]

```
// コンソールエラーの例:
Uncaught TypeError: Cannot read properties of undefined (reading 'id')
    at UserForm.handleSubmit (UserForm.tsx:42)
```

### 関連ファイル（推測）

[バグの原因として考えられるファイル・コードの場所（任意）]

### Engineerへのコメント

[修正時に参考になる情報・注意点]
```

## 重大度の判定基準

### High（リリースブロック）
- 主要機能が完全に使えない
- データ損失・破損が発生する
- セキュリティ上の問題（XSS・認証バイパス等）
- アプリケーションがクラッシュする

### Medium（リリース前に修正推奨）
- 機能は使えるが、一部の操作で問題が発生する
- 代替手段でなんとかなるレベル
- エラーメッセージが不正確・不親切
- パフォーマンスが基準値を超える

### Low（次フェーズで修正）
- 見た目の軽微な崩れ
- 誤字脱字・表記ゆれ
- 一部のブラウザ・デバイスでのみ発生する軽微な問題

## バグ対応フロー

```
1. Testerがバグを発見する
2. バグ報告書を作成する（上記テンプレートを使う）
3. Engineerへ差し戻す（重大度を明記）
4. Engineerが修正する
5. Engineerが修正内容をTesterへ報告する
6. Testerが再テストを実施する
7. 解消確認 → クローズ / 未解消 → 手順4に戻る
```
