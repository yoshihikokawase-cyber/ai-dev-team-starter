# Skill: 要件定義書を書く（write-spec）

## このスキルの目的

要望・課題を受け取り、実装可能な仕様書を作成します。
BAエージェントの出力形式に従い、`docs/requirements/` に保存します。

---

## 使い方

```
skills/write-spec/SKILL.md を読んで、以下の要望を仕様書にしてください:
[要望の内容]
仕様書は docs/requirements/[機能名].md として保存してください。
```

---

## 仕様書テンプレート

以下のテンプレートをコピーして仕様書を作成してください:

```markdown
# 仕様書: [機能名]

作成日: YYYY-MM-DD
作成者: Business Analyst Agent
ステータス: Draft / Review中 / 承認済み

---

## 背景・課題

[なぜこの機能が必要か。現状のどんな課題を解決するか]

## 目的

[この機能で達成したいゴール。KPIや成功指標があれば記載]

## 対象ユーザー

- ユーザー種別: [例: ログイン済みユーザー / 管理者 / 訪問者]
- 利用シーン: [どんな状況で使うか]

## ユースケース

### UC-01: [ユースケース名]
- アクター: [誰が]
- 事前条件: [どんな状態で]
- 基本フロー:
  1. [手順1]
  2. [手順2]
  3. [手順3]
- 代替フロー:
  - [異常ケース1]: [起きること・対処]
- 事後条件: [完了後の状態]

## 機能要件

- [ ] FR-01: [要件1]
- [ ] FR-02: [要件2]
- [ ] FR-03: [要件3]

## 非機能要件

- パフォーマンス: レスポンスタイム X秒以内 / 同時接続 X人
- セキュリティ: [認証方式・データ保護要件]
- アクセシビリティ: [WCAG2.1 レベルAA準拠など]
- 対応ブラウザ: Chrome最新版 / Firefox最新版 / Safari最新版 / Edge最新版
- 対応デバイス: Desktop / Tablet / Mobile

## 受け入れ条件

- Given [前提条件], When [操作], Then [期待結果]
- Given [前提条件], When [操作], Then [期待結果]
- Given [前提条件], When [操作], Then [期待結果]

## スコープ外（今回含めないもの）

- [除外事項1]: 理由 [なぜ今回は不要か]
- [除外事項2]: 理由 [なぜ今回は不要か]

## 未確認事項

- 未確認: [不明な点] → 確認先: [誰に確認するか]

## 関連ドキュメント

- [docs/product/vision.md](../product/vision.md)
- [docs/architecture/overview.md](../architecture/overview.md)
```

---

## 記入のコツ

### 受け入れ条件の書き方
- **良い例**: `Given ログイン済みユーザーが登録フォームを開いている, When 有効なメールアドレスとパスワード(8文字以上)を入力して送信する, Then アカウントが作成されトップページにリダイレクトされる`
- **悪い例**: `ちゃんと登録できること`

### 機能要件の粒度
- **良い例**: `メールアドレスはRFC5322形式で検証し、不正な場合はエラーメッセージを表示する`
- **悪い例**: `入力チェックをする`

### スコープ外の明示
曖昧に「後でやる」ではなく、「今回は○○しない。理由：△△」と明記することで、
実装中の認識ズレを防ぎます。
