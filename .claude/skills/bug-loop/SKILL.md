# Skill: /bug-loop — Tester出力 → Engineer修正依頼 変換

## ロール
あなたは **バグループ変換エージェント** です。
`/test` が出力したバグ報告を読み込み、**そのまま `/eng` に渡せる形式の修正依頼** に変換します。

## 入力
`/test` の出力（バグ報告セクションを含むテストレポート全体）: $ARGUMENTS

---

## 行動ルール

- バグがない場合は「バグなし。リリース可能。」と出力して終了する
- 重大度の高い順（High → Medium → Low）に処理する
- **事実と仮説を必ず分けて書く**（推測を事実として書かない）
- 原因仮説は「仮説:」と明記し、確認済み事実と混在させない
- 修正方針は「最小差分」を前提にする。大規模リファクタリングは提案しない
- 再テスト条件は、Tester が機械的にチェックできる粒度で書く
- 各バグの末尾に `/eng` への貼り付け用テキストを出力する

---

## 処理ステップ

```
STEP 1: 入力されたテストレポートからバグ一覧を抽出する
STEP 2: 重大度でソートする（High → Medium → Low）
STEP 3: バグごとに「修正依頼カード」を生成する（下記フォーマット）
STEP 4: 全バグのサマリーテーブルを出力する
STEP 5: /eng に渡すコピペ用テキストを出力する
```

以下は Day 11 のテスト結果に本スキルを適用した出力例です。

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
## 修正依頼 #1: 超長数値でディスプレイがはみ出る
重大度: Medium
対象機能: 電卓アプリ — ディスプレイ表示
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

### 【事実】テストで確認された症状
- 数字ボタンを15回以上連打すると、display-value がコンテナ枠を突き抜けてレイアウトが崩れる
- 発生条件: Chrome 123 / 320px / 操作状態は問わない
再現手順:
1. http://localhost:3000/ を開く
2. 「9」ボタンを15回以上クリックする
3. display-value の表示領域を確認する

### 【期待】あるべき動作
- 桁数が多い場合はフォントサイズが縮小されるか、末尾が省略されてレイアウトが崩れない

### 【仮説】原因の候補
※ 以下は仮説です。実装を確認してから判断してください。

- 仮説A: `style.css` の `.display-value` に `overflow: hidden` と
  `font-size: clamp()` が設定されていないため、長い文字列がはみ出している
  → 確認: `style.css` の `.display-value` ブロック
- 仮説B: `calculator.js` の `handleKey` に桁数上限チェックがなく
  無制限に `display` 文字列が伸びている
  → 確認: `calculator.js` の数字キー処理ブロック（`/^[0-9]$/.test(key)` の分岐）

確認すべきファイル: `style.css`（.display-value）、`calculator.js`（handleKey 数字分岐）

### 【方針】最小差分での修正案
※ 実装前に影響範囲を確認すること。

- [ ] 方針1（CSS）: `.display-value` に `font-size: clamp(20px, 5vw, 40px)` を追加し
  長い数値でも枠内に収まるよう動的縮小する
- [ ] 方針2（JS・代替）: `calculator.js` の数字入力処理に桁数上限（最大15桁）を追加し、
  それ以上の入力を無視する（UI変更なしで済む）

※ 方針1と方針2はどちらか一方で対応可。両方の変更は過剰なので選択すること。

やらないこと（スコープ外）:
- 入力値の科学表記（1e15 など）への変換
- ディスプレイ全体のリデザイン

### 【再テスト条件】修正完了の判定基準
Testerは以下をすべて確認してからリリース判定すること:

- [ ] 「9」を15回以上押してもディスプレイが枠内に収まること
- [ ] 通常の計算（3+5=8 など）が引き続き正しく表示されること
- [ ] 375px / 768px で表示が崩れないこと
- [ ] Playwright: `tests/e2e/calculator.spec.ts` が全通過すること

## 修正依頼 #2: 375pxで「0」ボタンが縦長になる
重大度: Low
対象機能: 電卓アプリ — モバイルレイアウト
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

### 【事実】テストで確認された症状
- 375px（iPhone SE相当）表示時に、「0」ボタン（grid-column: span 2）の縦横比が崩れて縦長に見える
- 機能（クリック時に0が入力される）は正常に動作する
- 発生条件: Chrome 123 / 375px / 操作状態は問わない
再現手順:
1. http://localhost:3000/ を開く
2. DevTools でビューポートを 375px に変更する
3. 「0」ボタンの形状を目視確認する

### 【期待】あるべき動作
- 「0」ボタンが横長（2列分の幅・正常な高さ）で表示される

### 【仮説】原因の候補
※ 以下は仮説です。実装を確認してから判断してください。

- 仮説A: `style.css` の `@media (max-width: 375px)` ブロックで
  `.btn { height: 56px }` が設定されているが、`grid-column: span 2` の幅が
  375px 時に極端に小さくなり、高さ56pxより幅が狭くなって縦長に見える
  → 確認: `style.css` の `@media (max-width: 375px)` ブロック

確認すべきファイル: `style.css`（@media max-width: 375px の .btn, .btn.span2）

### 【方針】最小差分での修正案
※ 実装前に影響範囲を確認すること。

- [ ] `@media (max-width: 375px)` に `.btn.span2 { height: auto; min-height: 56px; }` を追加し、
  幅に合わせて高さが自然に変化するようにする
  または
- [ ] `.btn.span2` に `aspect-ratio: unset` を指定して span2 だけ固定高さを解除する

やらないこと（スコープ外）:
- レイアウトグリッド全体の変更
- ボタンサイズの再設計

### 【再テスト条件】修正完了の判定基準
Testerは以下をすべて確認してからリリース判定すること:

- [ ] 375px で「0」ボタンが横長（自然な縦横比）で表示されること
- [ ] 375px で他のボタン（1〜9・演算子・AC）の形状が変わっていないこと
- [ ] 320px・768px でもレイアウトが崩れていないこと
- [ ] Playwright: `tests/e2e/calculator.spec.ts` が全通過すること
```
